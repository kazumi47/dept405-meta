# LLM評価サイクル（Evaluation Cycle for Ambiguous Domains）

正解が事前に存在しない領域（要約品質・関連性判定・人事評価・創作など）で、人間の判断を正解データ化し、機械の判定を校正し、委譲の可否を数値で決めるための型。
出典: DEPT.405「友人・政治まさはる」（2026-07）で人手評価41件→Judge校正→ブラインドA/B×2ラウンドまで一巡した実証プロセス。

## 0. 原則

1. **委譲には基準が先。** AIに判定・選別を任せるのは、人間の正解データとの一致率が事前に決めた合格ライン（例: 80%）を超えてから。未達なら人間継続——それは失敗ではなく校正が仕事をした状態。
2. **生成が安くなるほど、検証が高くなる。** もっともらしい出力は無限に作れる。価値は「正しさの保証」（評価・出典・監査可能なログ）に移る。
3. **測定は目標化した瞬間に歪む（Goodhartの法則）。** 利用量・件数などの投入指標をKPI化しない。測るなら委任と判断の質、出力側の証拠。
4. **正解が本人の実感にしかない領域では、採点＝正解の定義行為。** 採点者の「なんとなく」を軽視しない。溜めて言語化したものがルーブリックになる。

## 1. 標準工程（7段）

1. **データプロファイリング**: 対象の件数・分布を測る。件数は「量」であって「意味」ではない——件数だけで計画しない。
2. **サンプリングEDA**: 各セルから数十件サンプル→埋め込み→クラスタリング→LLMがクラスタにラベル（一言＋2文説明＋代表例）。キーワード起点の目視より、クラスタ起点は**想定外の意味**（誤爆・別文脈）を浮かび上がらせる。成果物は人間可読のレポート。
3. **ゴールデンセット構築**: 人間が20件以上採点（作法は§2）。
4. **ルーブリック化**: 採点コメントから判定基準を帰納する（人間が先に書くのではなく、実採点から抽出する）。
5. **LLM-as-a-Judge校正**: 安いモデルで判定→ゴールデンセットとの一致率＋混同行列をレポート。合格ライン未達ならモデル昇格 or 人間継続。
6. **（生成系なら）ブラインドA/B**: モデル・プロンプトの比較は目隠しで（作法は§4）。
7. **選好の蓄積**: 「両方合格だがこちらが好み」は選好ペアとして記録。将来のプリファレンス調整の原料。

## 2. ゴールデンセットの作法

- **verdictは三値＋棄権**: 「わかる（該当）／ちがう（非該当）／わからん（判定材料不足の棄権）」。
  - 「ちがう」は自信ある不合格、「わからん」は棄権。悩んだ末の「たぶん違う」は「ちがう」でよい。土俵に上がれなければ「わからん」＋理由。
  - **棄権枠は必須**。無理やり二択にすると採点が嘘になり、Judgeに濁った基準を教える。
- **両側を含める**: 「該当」だけでは判定器を校正できない。ただし両側を作るための嘘の「ちがう」は本末転倒——正直が最優先、正直に付ければ自然に散る。
- **コメント必須**: 「なぜその判定か」の一言がルーブリックの原料。迷いの言語化こそ最良の教材。
- **興味と関連は別勘定**: 面白いは免罪符にならない。関連の列を汚さず、面白い発見はコメントに宝メモとして残す。
- 棄権が集中したクラスタ・項目は、装置側の改善指示書（説明不足・サンプル汚染・材料不足の地図）。

## 3. Judge校正の作法

- **試験問題の設計を最初に疑う**: Judgeプロンプトが「厳密な事実対応」を要求すると、緩い関連を判定する課題では構造的に全滅する（実測: 一致率15%まで縮退した事例あり）。判定の定義はコンセプト文書の言葉に合わせる。
- **一致率＋混同行列を必ず出す**: どの誤り方をしたか（該当→非該当の反転が多い等）が改善の入口。
- **機械は棄権しない傾向がある**: 「わからん」を一度も予測しないJudgeは境界事例で過信する。人間の棄権はそれ自体が機械に無い情報。
- **ルーブリックの単純追加は悪化しうる**: 参照データ（属性一覧・中核テーマ等）を必要とする規則を、データ同梱なしでプロンプトに足すと、機械が適用できるのは「却下する規則」だけになり厳格側に偏る（片翼ルーブリック問題。実測: 55%→45%に悪化した事例あり）。規則には適用に必要なデータを必ず添える。

## 4. ブラインドA/Bの作法

- モデル対応表は**封印ファイル**に隔離し、採点完了まで誰も開かない。
- **事前登録ルール**: 開封前に「引き分けの場合どちらを採用するか」（例: 引き分けなら安価な方）を決めて記録する。結果を見てからルールを作らない。
- **評価者の独立性**: 採点前に他者（AI含む）の感想を評価者に見せない。
- 採点は半構造の4問程度: ①本体verdict ②文体・声 ③構成・流れ ④出典の抜き取り検査（各版2本開いて主張と原文の一致を確認）。
- 敗者から規則を抽出してプロンプトに焼く（例: 「交差の明示は冒頭と結びのみ」「並列直喩の禁止」）。ただし**制約は癖を止められるが品位は上げられない**——構成規則を守っても地力で負ける場合は素直にモデルを上げる。
- 再戦（プロンプト改善後の敗者復活戦）は**別の題材**で行う。同じ題材は記憶による汚染で公平でない。

## 5. グラウンディング

- 生成物の全主張はソースID（出典）に紐づける。根拠のない文には**[要検証]フラグ**を自動付与してレビュー行きにする——正直な自己申告は品質。
- 「出典IDが実在する」の機械チェックと「引用が意味的に妥当」は別物。後者の自動化は難しく、人間の抜き取り検査を残す。

## 6. モデル階層戦略

- **後段に人間の関所または検証器がある工程は安いモデル**（分類・候補生成・一次判定）。
- **最終読者に直接届く因果系の生成（要約・語り）は強いモデル起点**。
- 選定は信仰でなくブラインドA/B実測で決め、decisions.mdに決裁として記録する。

## 7. 運用の罠（実際に踏んだもの）

- **評価観点の表示は誘導バイアス**: 採点UIに観点を1つだけ表示すると採点がそれに引きずられる。無地か複数併記。
- **テンプレのデフォルト値残置事故**: 「合ってた／ズレた」形式の記入欄は未編集のまま提出されうる。選択式（ボタン）UIが構造的に防ぐ。**評価記録もデータ**——曖昧な記録から誤った結論が導出される。
- **摩擦の除去は検品緊張も除去する**: UIが滑らかになるほど人間の目は節穴化する。意図的な引っかかり（抜き取り確認等）を設計する。
- **コスト統制**: LLM実行の前に対象件数×概算単価を見積ログに出す。対象を先にテーブル化してから推論にかける。
