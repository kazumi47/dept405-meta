# 仕様駆動AI開発（Spec-Driven Development for AI Agents）

人間が「定義・検証・責任」を持ち、AIが「実装」を持つ分業を、文書で成立させる型。
出典: DEPT.405「友人・政治まさはる」開発（2026-07、企画から3バージョン竣工まで4日間）の実証済みプロセス。

## 0. 三原則

1. **整合性は文書が担保する。** 実装者（AIエージェント）は企画時の会話を知らない。会話で決めたことは文書に焼かれて初めて存在する。モデルを切り替えても文書が続きを保証する。
2. **人間は定義する・検証する・責任を持つ。AIは実装する。** 人間の仕事は「良い」の定義（要件・DoD）、出力の検証（レビュー・採点）、判断の記録（決裁）。細かい指示ではなく背景とゴールを渡し、推論過程を出させ、過程ごと検収する。
3. **実物が正。** 文書と実データ・実挙動が食い違ったら実物が正しい。差分は必ず decisions.md に記録して文書を追従させる。

## 1. 文書キット（5点セット）

| 文書 | 役割 | 必須構成 |
|---|---|---|
| concept.md | 製品の憲法（1枚） | これは何か／誰のためか／成功の定義／作らないもの／決裁記録 |
| requirements_vX.Y.md | 各バージョンの契約書 | 歩き方(§0)／スコープ(やる・やらない)／仕様／DoD／未決事項→決裁記録 |
| CLAUDE.md | 実装者への行動規約 | 役割分担／絶対要件／規約／実装順／**止まって聞く領域** |
| decisions.md | 意思決定の追記型台帳 | D番号／日付／決定／理由／決裁者。逸脱・例外・訂正も全部ここ |
| roadmap.md | 全体進行のliving document | ゴール像／駅一覧(完了記録つき)／並走トラック／申し送り |

- 要件書には「§0 この文書の歩き方」を置き、人間が読むべき節（スコープ・DoD・未決事項）を明示する。全読を強いる文書は読まれない。
- 用語は業界標準を正とする（学習教材と転用可能性のため）。俗称を使った場合は対応表を残す。

## 2. 起草→決裁方式

- ドラフトはAIが書いてよい。ただし**判断が要る箇所は「未決事項」として推し案＋対案＋理由を添えて人間に差し出す**。
- 人間の決裁後、未決事項の節を「決裁記録（日付つき）」に書き換えて確定版とする。
- 決裁の粒度: 名前・成功の定義・倫理的線引き・コストが絡む選択・スコープは必ず人間。実装手段の選択はAIの裁量でよい（事後にdecisions.mdへ）。

## 3. バージョン凍結運用

- 完了したバージョンの要件書は**凍結**。以後は編集せず、先頭にステータス行（完了日・D番号・commit）のみ追記。
- 次バージョンは新ファイルを起草→決裁→着工。実装中に要件書を書き換えると「何に対して検収したか」が消える。
- 派生物（レポート・エクスポート・図表）は手で直さない。**上流（データ・要件）を直して再生成する。**

## 4. 実装順とステージゲート

- **実装順0は常に環境監査。**「人間が手作業した環境は中途半端な可能性がある」前提で、構造検証・修復から入らせる。
- 原則: **工程の中身は自動、工程の間は人間が開ける。** 一気通貫の全自動モードは作らない。関所（人間確認）の位置を要件に明文化する——暗黙のままだと実装者は善意で全自動にする。
- CLAUDE.mdに「止まって聞く領域」を列挙（破壊的変更・費用発生・外部送信・本番実行など）。
- 指示文は**冪等**に書く: 「既に完了済みの項目はスキップし、その旨だけ報告して」を冒頭に。二重投入や先回り実装と衝突しない。

## 5. DoD（受け入れ基準）の設計

- 各項目は**検証可能**な文で書く（「〜が1コマンドで再構築できる」「〜がSQL一発で出る」）。
- 必ず入れる3種: ①再構築可能性（正本からのワンコマンド復元）②**ドッグフーディング**（作った本人が実際に使った証明を1件以上）③関所の実在確認（「全自動で貫通しないことの確認」）。
- DoD突合はAIにやらせ、completionレポートをファイルに残させる。ただし**完了報告と成果物は人間が突合する**（報告に書いてあることが成果物に無い事故は起きる。逆に、成果物にあるのに読み手が見落とす事故も起きる。どちらも実物を開けば30秒で決着する）。

## 6. アンチパターン（実際に踏んだもの）

- **呼称の二重帳簿**: 正式な進行表（実装順）と会話上の通称（フェーズ）が併存し混乱。進行表は一本化し、通称を使ったら対応をdecisions.mdへ。
- **暗黙の前提の未明文化**: 「うちは承認制の文化」は書かなければ存在しない。
- **派生物の手編集**: レポートを直接直すと次の再生成で消える・嘘になる。
- **ローカル設定のコミット**: `*.local.*` は個人環境設定。コミット対象から除外。
- **設計者の誤りは検証工程が拾う前提を崩さない**: 要件の桁数ミス・パーティション設計ミスは実装初日に実物との突合で発覚した。「実装順1=実レスポンス1件の確認」を省くと、この安全網が消える。
- **Vercel初回デプロイの罠**: 新規リンクしたプロジェクトへの初回デプロイは、`--prod`を
  付けなくても本番（production）として扱われ、エイリアスドメインに
  アクセス制限なしで公開されることがある（`dragon-ball-type`, D009関連の
  ガイドライン逸脱記録）。デプロイ関門（§4）では、初回デプロイ直後に必ず
  `curl -o /dev/null -w "%{http_code}" <URL>` 等で公開状態を実際に確認する
  ステップを入れる。「`--prod`を付けていないから安全」という思い込みで
  確認を省略しない。

## 7. 運用リズム

- 1工程≒1セッション。Conventional Commitsでこまめにコミット・push。
- テスト・検証の結果は端末ではなくファイル（reports/）に書き出させる。
- スコープ変更・優先度変更が起きたらroadmapを即改訂し、改訂自体をD番号で記録する。

## 8. 実践から得た技術パターン

- **Proプラン無しのアクセス制限（Basic認証 via proxy.ts）**: Vercelの純正
  Password ProtectionはProプラン（有料）が必要。無料の範囲で「自分専用・
  身近な人とだけ共有」程度のアクセス制限をかけたい場合は、Next.js 16の
  `proxy.ts`（旧`middleware.ts`）でHTTP Basic認証を自前実装する。認証情報は
  環境変数（例: `BASIC_AUTH_USER`/`BASIC_AUTH_PASSWORD`）に持たせ、
  未認証・誤認証には401を返す。ただしNext.js公式ガイドは「認証の唯一の
  防御としてproxy.ts/middlewareに頼らない」ことを推奨しており、機微な
  データを扱うアプリでは不十分（`dragon-ball-type`, D009）。
- **コンテンツ分離による著作権リスクの低減**: 特定作品のキャラ名・セリフ等
  著作物性のあるコンテンツをお題・素材として使うアプリは、そのデータを
  アプリ本体（コード）から切り離し、外部JSON等で差し替え可能な設計にすると
  良い。アプリ自体は汎用エンジンとして扱え、著作権のあるコンテンツは
  「ユーザーがローカルに用意する差し替えデータ」という位置づけに限定できる。
  同梱するサンプルデータ（組み込みデフォルト）はローカル限定・アクセス制限下
  での運用に留める（`dragon-ball-type`, D008）。

## 9. 学びの還流（スキル改訂の関所）

- 各バージョンの完了処理（DoD突合）に「スキル還流チェック」を含める:
  decisions.md の当該バージョン分を走査し、以下に該当する学びを候補として人間に提示する
  1. 一般化可能な失敗（このプロジェクト固有でない罠）→ 該当スキルへの追記案
  2. 既存スキルの記述と実態が食い違った箇所 → 改訂案
  3. 新しいノイズ型・アンチパターン → カタログへの追記案
- 提示は「候補＋差分案＋推し」の決裁形式。採否は人間。採用分は dept405-meta の
  該当SKILL.mdに追記し、出典（プロジェクト名・D番号）を残してpush
- retrospectives/ に1枚書く際も同チェックを実施
