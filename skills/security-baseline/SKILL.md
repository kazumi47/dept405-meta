# セキュリティ基本方針（Security Baseline for AI-Assisted Development）

バイブコーディングは「動くが安全ではない」コードを生みやすい構造を持つ。理由は3つ——
(1) AIは「動く」を優先し、セキュリティ制御は指示がなければ省く。
(2) 学習データ由来の古い・非推奨の実装パターンをもっともらしく出力する。
(3) 曖昧な指示は「通す」方向に倒し、検証・例外処理を省略しがち。

このスキルは、その構造的な穴を **実装時・リリース前・修正時** の3場面で塞ぐ型。
出典: セキュリティエンジニア講義資料（2026年・全42枚）を DEPT.405 の開発体制
（個人開発・Claude Code への実装委任・spec-driven-dev 運用）に翻訳したもの。
脆弱性分類は OWASP Top 10:2025 正式版（2026年1月）に準拠。

## 三原則

1. **AIは「安全な仕様」を推測しない。** 安全要件は明示して渡し、明示した要件でレビューする。
   「ログイン機能を実装して」ではなく「ログイン機能を実装。§1の必須条件をすべて守ること」。
2. **「動いた」は合格ではない。** 認証・認可・入力検証・秘密情報の4点が通って初めて出荷候補。
   とくに認可（本人確認・所有者照合）は、認証（ログイン確認）が実装済みでも抜けやすい——バイブコード最頻出の穴。
3. **チェックはAIが手伝う。出荷判断は人間がする。** 診断結果は「指摘一覧」ではなく
   「出荷判断パッケージ」（公開可／条件付き公開可／非推奨 の3択＋根拠）として人間に差し出す。
   出荷判断は spec-driven-dev でいう「止まって聞く領域」。AIが勝手に「問題なし」で進めない。

## 1. 実装ガードレール（常時適用・7箇条）

コードを書くとき、以下を指示の有無にかかわらず守る。プロジェクトの CLAUDE.md にも同じ7箇条を置く
（assets/claude-md-security-rules.md が転記用テンプレート）。

1. **認可チェックは必ずサーバ側で実施する。** フロントから渡された ID・role を信頼しない。
   リソース取得は DB クエリに「ログインユーザーが所有者か」の条件を含める（IDOR 対策）。
2. **SQL は ORM またはプリペアドステートメントを使用する。** 文字列連結での SQL 組み立て禁止。
3. **シークレット・API キーは環境変数から取得する。** コードへの直書き禁止。クライアント配布物に含めない。
4. **エラーメッセージは内部情報を含めない。** スタックトレース・SQL・内部パスをユーザーに表示しない。
   詳細はサーバ側ログへ。例外は握り潰さない（空 catch 禁止）——記録して安全側（fail-closed）に倒す。
5. **使用ライブラリは最新安定版を指定し、バージョンを固定する。** lockfile をコミットする。
6. **セッション・トークンの有効期限は設計段階で決める。** JWT はサーバ側で署名・期限を検証し、
   ログアウト時はサーバ側でセッションを無効化する。
7. **入力値は必ずサーバ側で検証・エスケープする。** HTML 出力はエスケープ（innerHTML 直代入禁止）、
   ファイルは MIME 検証＋ファイル名再生成、URL を受け取る機能は allowlist ＋内部 IP ブロック。

## 2. 出荷禁止アンチパターン（バイブコード5大穴）

以下を見つけたら、黙って通さず報告する。自分が生成するコードにも出さない。

| 穴 | 典型 | 検知の手がかり |
|---|---|---|
| ① 仮実装残存 | `// TODO: 認証追加` のまま本番へ | `grep -rn "TODO\|FIXME\|mock" src/` |
| ② 過剰権限 | 全 API を admin 固定・service_role で動かす | 権限判定コードの不在、固定 role |
| ③ 危険な近道 | 例外握り潰し・入力無検証・「暫定公開」 | 空の catch、検証なしの `req.body` 直使用 |
| ④ コピー実装由来 | 同じ脆弱なパターンが複数箇所に | 1箇所見つけたら同系統を grep（§4） |
| ⑤ 責務混在 | フロントで権限判定・API キーがクライアントに露出 | クライアントコード内の判定/キー |

LLM 機能を組み込むアプリ（チャット・RAG・エージェント）は別軸のリスクが加わる:
**Prompt Injection**（外部テキストによる誘導）・**Tool Abuse**（外部操作・権限逸脱）・
**RAG 経由の情報漏えい**（テナント混線）。LLM に「読む・実行する・送信する」権限を与えるほど
攻撃面は広がる。詳細: references/vulnerability-patterns.md §8。

## 3. リリース前チェックの手順

1. `scripts/quick_check.sh` を実行して機械チェック（TODO 残存・.env 追跡・audit・秘密情報パターン等）。
   ただし機械チェックで拾えるのは一部。**認可・IDOR・CSRF は grep では見つからない**。
2. references/release-checklist.md を開き、全項目を対象コードに照らして確認する。
   `!` 印の項目（認可・秘密情報系）は特に重要。
3. 結果は会話に流すだけでなく **ファイルに保存する**（例: `docs/security/check_YYYYMMDD.md`）。
   DEPT.405 ガイドライン「テスト結果はターミナルではなくファイルへ」に準拠。
4. 最後に **出荷判断** を3択＋根拠で人間に提出する:
   - **公開可** — Critical / High なし
   - **条件付き公開可** — 残リスクと運用条件を明記（例: Medium は次回改修、Low は運用で吸収）
   - **非推奨** — Critical / High が残存。何を直せば公開可になるかを添える
5. 残リスクと判断は decisions.md に記録する（spec-driven-dev の台帳と接続）。

## 4. 指摘修正の型

「指摘箇所を直す」ではなく「**脆弱な実装パターンをコードベースから消す**」。

1. 指摘箇所を確認し、優先度を2軸（影響の大きさ × 悪用の容易さ）で判定:
   Critical=即時（認証バイパス等）/ High=リリース前必須（IDOR・SSRF等）/
   Medium=次回改修（CORS 不備等）/ Low=運用で吸収可（ヘッダー漏れ等）。
2. **同系統コードを grep で全検索**（横展開）。バイブコードは同じパターンが複数箇所に重複しやすい。
   例: `grep -rn "req.params.id" src/ | grep -v "req.user.id"` — 本人確認なしの ID アクセスを検索。
3. 修正＋テスト。
4. **再確認条件を定義して提出**——「直ったと判断するテスト条件」を明文化する
   （どの入力・権限・環境で確認したか）。
詳細な書式と診断フロー全体: references/triage.md。

## 5. DEPT.405 スタック対応表

実装時、使っている技術に応じて以下を適用する。

| 領域 | ルール |
|---|---|
| Next.js / Vercel | `NEXT_PUBLIC_` 付き環境変数はクライアントに露出する——秘密には絶対付けない。認可判定は Route Handler / Server Action / Server Component 側で行い、middleware 単独やクライアント判定に頼らない |
| Supabase | 全テーブルで RLS を有効化しポリシーを明示（Security Advisor の警告ゼロが出荷条件）。`service_role` キーはサーバ専用——クライアント配布物・リポジトリに含めない |
| Neon + Drizzle | クエリビルダ経由は自動でパラメータ化される。`sql` テンプレートで生 SQL を書くときのみ注意（文字列連結禁止） |
| Cloudflare R2 / S3 系 | バケット公開設定を棚卸しする。配布は署名付き URL を基本とし、バケット直公開は明示的な決裁があるときだけ |
| Claude API 等の LLM | キーは環境変数・サーバ経由で呼ぶ（クライアントから直接叩かない）。ユーザー入力・取得ドキュメント・検索結果は全て不信入力として扱う。ユーザー毎のレート制限を入れる |
| LINE / Discord Webhook | 署名検証（`X-Line-Signature` 等）をサーバ側で必ず実施。検証なしの Webhook 受信口は「誰でも叩ける内部 API」 |
| BigQuery / GCP | サービスアカウントは最小権限。認証 JSON はリポジトリに入れない（.gitignore 確認） |
| 課金・チケット・利用回数 | 注文・チケット・残回数は必ずサーバ側で所有者照合と残高判定（IDOR・改ざん対策）。フロントの表示値を信頼しない |

## 注意

- Claude Code の設定キー（sandbox・permissions 等）はバージョンで変わる。適用前に
  `/permissions`・`/sandbox` と公式ドキュメント（https://code.claude.com/docs/en/permissions）で確認する。
- このスキルは「最小対策」の入口。設計・実装・設定・運用を組み合わせて、継続的にリスクを下げる。
  セキュリティは一度で完成しない——習慣として組み込む。
