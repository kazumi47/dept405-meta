# 診断とトリアージの型

診断結果は「指摘一覧」ではなく **「出荷判断パッケージ」**——「何が悪いか」だけでなく
「出荷できるか／何を条件に出荷するか」を人間が判断できる形にする。

## §1 診断から出荷判断までの7ステップ

| # | ステップ | 内容 | 成果物 |
|---|---|---|---|
| ① | 対象範囲の定義 | URL・API・リポジトリ・クラウド構成を固定する | 対象範囲定義書 |
| ② | 攻撃面の整理 | 公開資産・認証境界・外部連携を可視化する | 攻撃面整理シート |
| ③ | テクニカル診断 | 認証／認可／IDOR／CORS／CSRF／SSRF／Cloud／Supply Chain | テクニカル診断結果レポート |
| ④ | リスク評価 | Critical → High → Medium → Low で優先順位付け | リスク評価一覧 |
| ⑤ | 改修ガイド提供 | 再現手順・修正案・横展開観点 | 改修ガイド |
| ⑥ | 修正・再確認 | 修正後の再テスト・残留リスクの整理 | 再確認結果サマリ |
| ⑦ | 出荷判断 | 公開可／条件付き可／非推奨 の3択で判断（人間） | 出荷判断メモ |

個人開発の小規模リリースでは①②は release-checklist.md の §0 で兼ね、③〜⑦を回す。

## §2 優先度は2軸で判断する: 影響の大きさ × 悪用の容易さ

| 優先度 | 対応時期 | 代表例 | 判断 |
|---|---|---|---|
| Critical | 即時（本番停止も視野） | 認証バイパス | 攻撃が容易で影響も大きい |
| High | リリース前に必須 | IDOR・SSRF | 公開前に残すと実被害へ直結 |
| Medium | 次回改修で対応 | CORS 設定不備 | 条件付きで悪用可能 |
| Low | 運用での吸収も可 | ヘッダー設定漏れ | 影響が限定的 |

判断のポイント3つ:
- **影響**: 何が起きるか——情報漏えい／改ざん／停止
- **容易さ**: 認証の要否／公開範囲／再現性
- **到達人数**: 被害は1人か、全利用者か

## §3 横展開: 1箇所直したら、同系統を全部検索する

バイブコードは **同じパターンが複数箇所に重複しやすい**（コピー実装由来）。指摘箇所だけ直しても、
同じ穴が別の場所に残る。目標は「指摘箇所を直す」ではなく
「**脆弱な実装パターンをコードベースから消す**」。

手順:
1. 指摘箇所を確認する
2. 同系統コードを grep で全検索する
   - 例: `grep -rn "req.params.id" src/ | grep -v "req.user.id"` — 本人確認なしの ID アクセス
   - 例: `grep -rn "innerHTML" src/` — エスケープなしの HTML 出力候補
   - 例: `grep -rn "findById\|findByPk" src/` — 所有者照合の有無を1件ずつ確認
3. 修正 ＋ テスト
4. 再確認条件を定義して提出する

## §4 レポート3点書式（指摘1件ごとに）

```markdown
### [優先度] 指摘タイトル（OWASP 分類: A0x）
① 再現条件: どの入力・権限・環境で起きるか
   例: 未ログインでも GET /api/orders/{他人のID} が 200 を返す
② 影響と優先度: 何が漏れる／壊れるか、いつ直すか
   例: 全ユーザーの注文情報（氏名・住所）が漏えい → High、リリース前必須
③ 修正・再確認条件: 直ったと判断するテスト条件
   例: DB クエリに userId 条件を追加。他人の ID 指定で 404、自分の ID で 200 を確認
横展開: 同パターンの検索結果（対象ファイル一覧）と修正状況
```

3点が揃っていない指摘は「直ったかどうか判定できない」——必ず埋める。
全件のレポートに release-checklist.md の出荷判断書式を付けて人間に提出する。
