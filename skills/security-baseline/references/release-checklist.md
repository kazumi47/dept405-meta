# リリース前チェックリスト

リリース・デプロイ・公開の前に全項目を対象コード・設定に照らして確認する。
`!` は特に重要度が高い項目（バイブコードで実被害に直結しやすい）。
結果はファイル（例: `docs/security/check_YYYYMMDD.md`）に保存し、末尾の出荷判断書式で締める。

## 0. 攻撃面の可視化（最初にやる）

**「知らない API が公開されている」がいちばん危険。** 公開資産の一覧がなければ、診断対象から漏れる。

- [ ] 外部公開しているもの一覧: 公開 API・管理画面・ストレージ（S3/R2 等）
- [ ] 内部連携の一覧: Webhook 受信・DB・CDN・CI/CD・LLM 連携
- [ ] ! API 一覧を把握しているか（ルーティング定義から機械的に列挙する）
- [ ] ! 管理画面・Admin パネルにアクセス制限はあるか
- [ ] 外部公開バケットはないか
- [ ] CI/CD トークンは適切に保護されているか
- [ ] LLM への入力源（ユーザー入力・Web 取得・DB・ファイル）を把握しているか

## 1. 認証チェック — 「誰か」をサーバ側で確認する

- [ ] ! JWT やトークンはサーバ側で署名・有効期限を検証しているか
- [ ] ! ログアウト時にセッションをサーバ側で無効化しているか
- [ ] パスワードは bcrypt 等で適切にハッシュ化されているか
- [ ] ログイン失敗回数に制限があるか（ブルートフォース対策）
- [ ] パスワードリセットのトークンに有効期限があるか

## 2. 認可チェック — 「何をしてよいか」をサーバ側で確認する

- [ ] ! API はログイン確認だけでなく「本人確認」（所有者照合）もしているか
- [ ] ID を連番・推測可能な形で外部に渡していないか
- [ ] ! 管理者機能は role 確認をサーバ側で行っているか
- [ ] ! フロントエンドだけで権限判定をしている箇所はないか

## 3. CORS — ブラウザが許可／拒否を判断

- [ ] `Access-Control-Allow-Origin` が `*`（ワイルドカード）や Origin 無条件反射になっていないか
- [ ] 許可ドメインを明示的にリストアップしているか
- [ ] credentials とワイルドカード相当を組み合わせていないか

## 4. CSRF — ログイン状態を悪用させない

- [ ] 状態変更 API に CSRF トークンが設定されているか（Cookie 認証の場合）
- [ ] Cookie に SameSite 属性が設定されているか
- [ ] 重要操作（送金・削除・課金等）に再認証を要求しているか

## 5. SSRF — URL 取得機能を入口として扱う

- [ ] （確認）ユーザーから URL を受け取る機能があるか（画像取得・プレビュー・Webhook・OGP）
- [ ] 許可するドメインを allowlist で管理しているか
- [ ] 内部ネットワーク・メタデータ API（169.254.169.254）へのアクセスをブロックしているか

## 6. 入力・出力

- [ ] DB クエリはすべて ORM またはプレースホルダを使っているか
- [ ] HTML に出力する値はエスケープされているか（innerHTML 直代入なし）
- [ ] ファイルアップロードの MIME タイプをサーバ側で検証しているか
- [ ] ファイル名を受け取ったまま使っていないか（パストラバーサル——保存名は再生成）

## 7. バイブコード固有

- [ ] ! TODO / mock / debug route が本番コードに残っていないか
- [ ] ! フロントエンドで権限判定をしている箇所はないか（§2 と重複確認——最頻出のため二重に見る）
- [ ] ! 秘密情報（API キー等）がクライアント配布物に露出していないか（`NEXT_PUBLIC_` 誤用含む）
- [ ] 例外を握り潰しているコードはないか（try/catch で何もしていない・成功だけ返す）

## 8. Secrets / Cloud / パッケージ

- [ ] Secrets は Secrets 管理サービス（ホスティングの環境変数等）に格納しているか（.env のままでないか）
- [ ] .env ファイルが .gitignore に含まれ、Git 履歴に載っていないか
- [ ] S3 / R2 等のバケットが意図せず公開設定になっていないか
- [ ] npm audit / pip-audit で Critical / High がゼロか
- [ ] Supabase 使用時: 全テーブル RLS 有効＋ Security Advisor 警告ゼロか

## 9. デバッグ・監視（設定・運用）

- [ ] デバッグモードが本番で無効化されているか
- [ ] 詳細なエラーメッセージ（スタックトレース等）が本番で非表示になっているか
- [ ] ログが適切に設定・監視されているか（異常を検知できるか）
- [ ] デバッグ用エンドポイントが本番に残っていないか
- [ ] 外部スクリプト（CDN 経由 JS）の棚卸しをしたか（不要なら削除・SRI・CSP）

## 出荷判断書式（レポート末尾に必ず付ける）

```markdown
## 出荷判断
判定: 公開可 ／ 条件付き公開可 ／ 非推奨   ← 3択で明記
根拠: （Critical/High の有無、未確認項目）
残リスク:
- [Medium] 例: CORS の許可ドメインが開発用を含む → 次回改修で対応
- [Low]    例: セキュリティヘッダー一部未設定 → 運用で吸収
条件（条件付き公開可の場合）: 公開してよい条件・期限・監視方法
再確認条件: 何を直して、どのテストが通れば「公開可」に上がるか
```

判定は人間が行う。AI はこの書式まで埋めて提出し、判断を仰ぐ。
