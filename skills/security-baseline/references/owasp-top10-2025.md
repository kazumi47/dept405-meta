# OWASP Top 10:2025 × 最小対策

OWASP Top 10:2025 正式版（2026年1月リリース、RC は 2025年11月発表）準拠。
2021年版からの主な変化: **A02 セキュリティ設定ミスが #5→#2 に上昇**、
**A03 ソフトウェアサプライチェーンが新設**（旧「脆弱・古いコンポーネント」を拡張）、
**A10 例外条件の不適切な処理が新設**、**SSRF は A01 に統合**。

## 対応表

| # | カテゴリ | 主なリスク | AI がやりがちなこと | 最小対策 |
|---|---|---|---|---|
| A01 | アクセス制御の不備 | IDOR・role 回避・他人データへのアクセス・SSRF（統合） | 認可チェックを省略して生成／user_id をフロントから信頼して DB 検索 | サーバ側で認可——所有者・role を毎回照合。フロントからの ID 指定を信頼しない |
| A02 | セキュリティ設定ミス ↑↑ | debug 残存・CORS 全許可・公開ストレージ | デバッグモードをそのまま本番へ／.env を Git にコミット／CORS を過度に広く許可 | 本番設定を deny-by-default 化し、出荷前に棚卸し。デプロイ前チェックリスト必須化・.gitignore に .env |
| A03 | ソフトウェアサプライチェーン NEW | npm/CDN/MCP の汚染・認証情報窃取 | 外部パッケージ・生成物を無検証で採用 | lockfile 固定・監査（audit）・署名検証・信頼元限定（詳細: supply-chain-and-agents.md） |
| A04 | 暗号化の失敗 | 平文保存・弱いハッシュ・TLS 不備 | 古い暗号方式・自作ハッシュをもっともらしく出力 | TLS・鍵管理・暗号アルゴリズムの棚卸し。パスワードは bcrypt 等の実績あるハッシュ |
| A05 | インジェクション | SQL・XSS・コマンドへの不正入力混入 | 文字列連結で SQL を組み立てる／出力エンコードなしで HTML へ埋め込む | パラメータ化（ORM／プリペアドステートメント）・出力エスケープ・入力検証 |
| A06 | 安全でない設計 | 有効期限・再認証・レート制限の欠落 | 機能要件だけで実装を積み上げる | 脅威分析とセキュリティ要件・濫用ケースを設計段階で定義 |
| A07 | 認証の失敗 | 署名未検証・弱いセッション・リセット穴・ブルートフォース | 認証を自作し、失効・期限を省略 | 実績ある認証基盤（既製ライブラリ）＋ MFA ＋ レート制限。署名・期限・失効を検証 |
| A08 | ソフトウェア／データ整合性の失敗 | CI/CD・更新物・成果物の改ざん | 外部コードや生成物を無検証で採用 | 署名検証・保護されたパイプライン・最小権限 |
| A09 | ロギング／アラートの失敗 | 攻撃・エラー・不正操作を検知できない | 例外を握り潰し、成功だけ返す（空の catch） | 構造化ログ・監視・アラートへ接続。監査証跡を整備 |
| A10 | 例外条件の不適切な処理 NEW | 情報漏えい・fail-open・復旧不能 | 異常系を省略し、想定外で安全側に倒れない | fail-closed・一般化エラーメッセージ・上限値を設計 |

## バイブコーディングが直撃するのは A01・A02・A03・A05

この4カテゴリは AI 生成コードの構造的な癖（認可省略・設定放置・依存無検証・連結クエリ）と
直接対応する。実装時は §上表の「AI がやりがちなこと」を自分の出力に対するチェック項目として使う。

## 使い方

- 診断レポートの分類・優先度づけにこの番号体系を使う（例: 「A01: /orders/:id に所有者照合なし」）
- 最小対策は入口。設計・実装・設定・運用を組み合わせて継続的にリスクを下げる
