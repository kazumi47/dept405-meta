# DEPT.405 開発ガイドライン

> バイブコーディング × Spec-Driven Development（SDD）  
> 2026 年版・ノンエンジニア開発者向け

---

## 0. このガイドラインの目的

- 複数プロジェクトを並走しながら、**ドキュメントと実装の乖離を防ぐ**
- 「実装中の流れでグダグダになる」を防ぐ
- スキルが確実に高まる進め方を標準化する
- 1 人開発でも、後から自分・他人が見て分かる構造を保つ

---

## 1. 基本思想：Spec-Driven Development（SDD）

### 従来のやり方（避けたい）

```
実装 → 動作確認 → 「あ、これ違う」 → 修正 → 要件定義書も更新（よく忘れる）
```

### SDD 方式（採用する）

```
「あ、これ違う」と気づく
  ↓
まず docs/requirements.md を更新
  ↓
AI に「最新の requirements.md に従って実装」と指示
  ↓
実装
  ↓
動作確認
```

**ポイント**: ドキュメントが「コードの設計図」であり「Single Source of Truth」になる

## 2. プロジェクトの標準ファイル構造

```
project/
├── CLAUDE.md              # Claude Code が読み込む開発ルール
├── README.md              # プロジェクト概要、セットアップ手順
├── docs/
│   ├── requirements.md    # 要件定義
│   ├── design.md          # ER 図、画面遷移、アーキテクチャ
│   ├── decisions.md       # ADR（重要な意思決定の履歴）
│   ├── tasks.md           # 実装タスクの一覧（Phase 別）
│   └── roadmap.md         # v1.0 / v1.1 / v2.0 ... の全体像、着手タイミング
├── src/                   # ソースコード
├── tests/                 # テストコード
├── .gitignore
└── package.json
```

### 各ドキュメントの役割

| ファイル | 役割 | いつ更新するか |
|---------|------|---------------|
| **CLAUDE.md** | 開発ルール、必読ドキュメント、技術スタック | プロジェクト開始時、ルール変更時 |
| **requirements.md** | 機能要件、非機能要件、UX 方針 | 要件変更時 |
| **design.md** | ER 図、画面遷移、アーキテクチャ | 設計変更時 |
| **decisions.md** | なぜその技術を選んだか、なぜその設計にしたか | 重要な意思決定時 |
| **tasks.md** | 現在の Phase、次のタスク | 各タスク開始・完了時 |
| **roadmap.md** | v1.0 / v1.1 / v2.0 の全体像、着手タイミング、完了条件 | バージョン計画の変更時 / **Phase の進捗が変わったとき** |
| **README.md** | プロジェクト概要、セットアップ、起動方法 | 環境変更時 |

## 3. プロジェクトの進め方（Step 別）

> **注意**：ここでの「Step」は開発全体の流れを指す。  
> プロジェクト内の実装区切りは「Phase」と呼んで区別する。  
> （例：プリント全録の Phase 1 は「基本 UI 実装」を指す）

### フェーズとステップの関係（重要）

**ウォーターフォールではなく、イテレーション（アジャイル的）**

各 Phase の中で、Step 0〜4 のミニサイクルを回す：

```
Phase 1（MVP）
  ├── Step 0: 要件定義（MVPの要件を固める）
  ├── Step 1: 設計（MVPの設計）
  ├── Step 2: 実装計画
  ├── Step 3: 実装
  └── Step 4: テスト・リリース
  ↓
Phase 2（次の機能群）
  ├── Step 0: 要件定義（追加機能の要件）
  ├── Step 1: 設計（差分設計）
  ├── Step 2: 実装計画
  ├── Step 3: 実装
  └── Step 4: テスト・リリース
  ↓
Phase 3 ...
```

### MVP を最小限にする思想

- **最初の Phase 1 = MVP** は、本当に必要最小限の機能だけ
- 完璧を目指さない
- 早く動くものを作って、使ってみて、次の Phase で拡張
- 「もし、これが Phase 1 で十分なら、Phase 2 はやらなくていい」くらいの気持ち

### Phase の分け方の例

```markdown
Phase 1: MVP（最小限で動く）
Phase 2: コア機能の拡張
Phase 3: UX 改善
Phase 4: クラウド同期、認証
Phase 5: マルチプラットフォーム対応
```

---

### Step 0：要件定義

1. **コンセプトメモ**を書く（紙のような自由フォーマット）
2. **requirements.md v0.1** を作成
3. **クリティカルレビュー**：自分で読み直す、ChatGPT などで違う AI と議論
4. **requirements.md v1.0** に修正

### Step 1：設計

1. **design.md** を作成
   - ER 図（Mermaid）
   - 画面遷移図
   - アーキテクチャ図（コンポーネント構成、データフロー）
2. **decisions.md** を作成
   - 技術スタック選択理由
   - 重要な設計判断

### Step 2：実装計画

1. **tasks.md** を作成
   - **Phase 別**にタスクを分割（ここで初めて Phase が登場）
   - 各タスクを 1-2 時間で終わるサイズに

### Step 3：実装

1. **CLAUDE.md** を作成（開発ルール）
2. Claude Code セッション開始
3. 1 タスクずつ実装、テスト、コミット
4. Phase 1 → Phase 2 → ... と進める

#### 実装前のチェックリスト（重要）

**実装に入る前に、必ず以下を確認する：**

1. **システム日付の当月時点のベストプラクティスを検索する**
   - 「最新」は曖昧で、AI の訓練データのカットオフを指してしまうことがある
   - **必ず「システム日付の当月（YYYY 年 MM 月）時点」と明記して検索**
   - 例：「2026 年 5 月時点での React のベストプラクティス」
   - 公式ドキュメントで現状を確認

2. **API より安定したコードを優先**
   - 外部 API を叩く前に、ローカルで実装できないか検討
   - 「ローカル処理 vs API 呼び出し」の比較（コスト、レイテンシー、依存性）
   - API を使うなら、無料枠・上限・将来の料金を確認

3. **公開ライブラリを探す**
   - 車輪の再発明を避ける
   - GitHub、npm で類似ライブラリを検索
   - 確認項目：
     - 無料か（ライセンス）
     - メンテされているか（最終更新日）
     - ダウンロード数・スター数（信頼性）
     - 自分のユースケースに合うか
   - 採用したら、decisions.md に「なぜそのライブラリを選んだか」を記録

### Step 4：テスト・リリース

1. 重要ロジックの単体テスト
2. ユースケースごとの手動テスト
3. デプロイ（Vercel / Google Play など）

## 4. セッション運用ルール

### セッション開始時

**毎回、必ず以下のプロンプトから始める：**

```
[プロジェクト名] を開きました。

以下のドキュメントを順に読み込んで、現状を把握してください：
1. CLAUDE.md（開発ルール）
2. docs/requirements.md（要件定義）
3. docs/design.md（設計）
4. docs/decisions.md（過去の意思決定）
5. docs/tasks.md（現在のタスク）

読み終わったら、現在の Phase と次のタスクを教えてください。
それから作業に入ります。
```

### 実装中のルール

1. **要件変更が必要だと感じたら、まず docs/requirements.md を更新する**
   - その後で実装に入る
2. **新しいライブラリを使う時は、docs/decisions.md に理由を書く**
3. **重要なロジックには単体テストを書く**（明示的に「テストも書きます」と AI に依頼）

### セッション終了時

**必ず以下をチェック：**

- [ ] 変更内容は docs/requirements.md に反映されたか
- [ ] 設計判断は docs/decisions.md に追記したか
- [ ] tasks.md の完了タスクをマークしたか
- [ ] テストが書けるロジックには書いたか
- [ ] git commit して、コミットメッセージは明確か

## 5. テスト戦略

### Claude Code の実態

**Claude Code は明示的に指示しないと、単体テストを書きません。**

「テストも書いて」「TDD で進めて」と指示する必要がある。

### テストの 3 段階

| 段階 | 対象 | 担当 |
|------|------|------|
| **単体テスト** | 重要なロジック（OCR、データ変換、計算） | AI に依頼 |
| **統合テスト** | コンポーネント連携、データフロー | 重要なものだけ AI に依頼 |
| **手動テスト** | ユースケース、UX、見た目 | 自分でやる |

### テスト計画の書き方

実装前に、「テストケース」をリストアップする：

```markdown
## OCR キーワード判定のテストケース

- [ ] 「計算ドリル」と書かれた画像 → 「学習」カテゴリ
- [ ] 「学級だより」と書かれた画像 → 「お知らせ」カテゴリ
- [ ] 「参観のお知らせ」と書かれた画像 → 「破棄予定」カテゴリ
- [ ] 文字が読めない画像 → 「未分類」カテゴリ
- [ ] 複数キーワードが混在する場合 → 優先度に従う
```

これを AI に渡して「これらのテストを書いて」と指示する。

### テスト結果は必ずファイルに残す（重要）

Claude Code でテストを実行すると、デフォルトでは結果がターミナルに表示されるだけで **後から確認できない**。これは AI 駆動開発の落とし穴。

**対策：テスト結果を必ずファイルに出力する設定にする**

#### Vitest / Jest（Node.js プロジェクト）の例

`package.json` の test スクリプトで `tee` を使う：

```json
{
  "scripts": {
    "test": "vitest run 2>&1 | tee test-results.txt",
    "test:watch": "vitest"
  }
}
```

これで `npm test` を実行すると、ターミナルに表示しつつ `test-results.txt` にも書き出される。

#### .gitignore に追加

テスト結果ファイルはコミットしないのが一般的：

```
# .gitignore
test-results.txt
coverage/
```

#### なぜ必須か

- ユーザーが後から「テスト本当に通った？」を確認できる
- AI のターミナル出力は流れて消えてしまう
- 失敗時のスタックトレースも記録に残る
- セッションを跨いだ振り返りができる

**新規プロジェクトでテストを書く時は、最初にこの設定を入れる**ことをルール化する。

## 6. ドキュメント書き方のコツ

### requirements.md

- **「何を作るか」を箇条書きで詳細に**
- フォーマットは Markdown
- セクション番号で構造化
- バージョン番号をつける（v0.1, v0.2, v1.0...）

### design.md

設計書の最大の罠は **「最初からシステム内部の言葉で書き始めること」**。これを避けるため、以下の順序で書く：

#### 1. ユーザー視点から始める

冒頭に「このアプリが何をするか」を、**技術用語なしで 3-5 行で説明**する。  
処理の流れも、人間の言葉で 3 ステップくらいに要約する。

#### 2. データ定義は、ユーザー側の言葉で

要件定義書に登場する用語（プリント、子ども、見たよマーク など）でデータ構造を説明する。

- 各データが「何を持つか」「何のためにあるか」を、自然な日本語で
- データ間の関係は、まずテキストツリーや自然言語で図解
- カテゴリや状態の値も、まず日本語の意味を表で示す

#### 3. システム内部へのマッピングは、独立セクションで

「ユーザー側の言葉 → システム内部の名前」の対応表を明示する。

例：
```
| ユーザー側の言葉 | システム内部の名前 |
|---|---|
| プリント | Print |
| 見たよマーク | isViewedByParent (boolean) |
| 確認待ち | pending |
```

ER 図、カラム名、Enum 値はこのマッピング表の後に登場させる。

#### 4. ER 図は Mermaid で

データのマッピングが完了した後に配置。リレーション名も日本語で（「持つ」「管理する」など）。

#### 5. 画面一覧と画面遷移図

画面名・対象ユーザー・Phase の表 + Mermaid または ASCII アート。

#### 6. 技術詳細は後半に

技術スタック、ディレクトリ構成、実装メモは、読み手の興味の順序に合わせて後半に。

#### なぜこの順序なのか

SI 業界の伝統的な設計手法では「概念モデル → 論理モデル → 物理モデル」の 3 段階で設計する。  
AI 駆動開発でも、これを飛ばすと、後で読み返した時に「何のためのデータだっけ」がわからなくなる。  
ユーザー視点（=要件定義書の言葉）を主軸にすることで、要件と設計の対応関係が崩れない。

### decisions.md（ADR）

各意思決定を以下の形式で記録：

```markdown
## [日付] [意思決定のタイトル]

### コンテキスト
なぜこの判断が必要だったか

### 検討した選択肢
- 選択肢 A：メリット / デメリット
- 選択肢 B：メリット / デメリット

### 決定
選択肢 X を選んだ

### 理由
なぜそれを選んだか

### 結果
その判断による影響、今後の見通し
```

### tasks.md

```markdown
## Phase 1：基本 UI 実装

- [x] 撮影画面のレイアウト
- [x] ガイド枠表示
- [ ] 連続撮影機能
- [ ] 子ども選択画面

## Phase 2：ローカル保存

- [ ] IndexedDB セットアップ
- [ ] サムネイル保存
- [ ] メタデータ保存
```

### 全ドキュメント共通：バージョン履歴の書き方

各ドキュメントの末尾には「バージョン履歴」を表形式で残す。  
ただし、Git のコミット履歴と二重管理にならないよう、**書き分けルール**を守る：

| 種別 | ドキュメントの履歴表 | Git のコミット履歴 |
|---|---|---|
| 細かい変更（誤字修正、表現調整など） | 書かない | 書く |
| 大きな節目（バージョン番号が上がるとき） | 書く（1 行） | 書く（詳細） |
| 構造変更・方針転換 | 書く | 書く（詳細） |

つまり、**「節目だけドキュメントに残す」**。日々の細かい変更は Git に任せる。

履歴表のフォーマット例：

```markdown
## バージョン履歴

| バージョン | 日付 | 内容 |
|---|---|---|
| v0.1 | 2026-05-19 | 初版作成 |
| v0.2 | 2026-05-19 | データセクションをユーザー視点で書き直し |
| v0.3 | 2026-05-20 | 子どもの権限拡張、フォルダ共有設定追加 |
```

## 7. グダグダ防止のテクニック

### 1. 「実装前に確認」を強制するプロンプト

```
実装に入る前に、docs/requirements.md のセクション 2.X を読んで、
現在の要件を要約してください。
それから実装を始めます。
```

### 2. 「変更検知」のチェックポイント

```
今、実装した内容は requirements.md と一致してますか？
ズレがあれば、まず requirements.md を更新してください。
```

### 3. セッション終了時の振り返り

```
本セッションでの変更を、以下のドキュメントに反映しましたか？
- requirements.md
- design.md
- decisions.md
- tasks.md

未反映があれば、今、更新してください。
```

### 4. ガイドライン逸脱の扱い（重要）

完全に厳格にすると、開発の楽しさが消える。  
でも、無視するとグダグダになる。  

そこで、**「警告するけど、強制はしない」** という方針を取る：

#### AI 側の役割

実装中、以下に気づいたら、必ずユーザーに確認する：

1. ドキュメントを更新せずに実装しようとしてる
   → 「先に requirements.md を更新しますか？それともこのまま進めますか？」

2. 検討せずに外部 API を叩こうとしてる
   → 「ローカルで実装できないか先に検討しますか？」

3. ライブラリ調査せずに自前実装しようとしてる
   → 「公開ライブラリを探してから決めますか？」

4. テストを書かずに重要ロジックを実装しようとしてる
   → 「単体テストを書きますか？」

#### ユーザー側の判断

- 「今回はこのまま行く」と言えば、AI はその判断を尊重する
- 「先にドキュメント更新する」と言えば、AI はそちらに切り替える
- どちらでも OK

#### 逸脱は記録する

ユーザーが「ガイドラインから外れる判断」をした場合、**decisions.md に記録する**。

```markdown
## [日付] ガイドライン逸脱：[内容]

### 状況
何を実装しようとしていたか

### ガイドラインの指示
本来はこうすべきだった

### 実際の判断
こうした

### 理由
なぜ逸脱したか（時間的制約、楽しさ優先、緊急性など）

### 振り返り
後で見直して、この判断は正しかったか
```

これで「ドタバタした楽しさ」も「後の振り返り」も両立できる。

## 8. プロジェクト管理：複数並走時のコツ

DEPT.405 では複数プロジェクトを並走している。各プロジェクトに：

- 独自の **CLAUDE.md**
- 独自の **docs/** フォルダ
- 独自の **tasks.md**

を持たせる。

セッションを切り替える時は、必ず：
1. 前のプロジェクトの tasks.md を更新
2. 新しいプロジェクトの CLAUDE.md と docs を再読み込み

これで「あれ？このプロジェクト、どこまでやったっけ」を防ぐ。

## 9. プロジェクトテンプレート（雛形）

新しいプロジェクトを始める時の標準セット：

```bash
# プロジェクト作成時
mkdir my-new-project
cd my-new-project
mkdir docs src tests
touch CLAUDE.md README.md
touch docs/requirements.md docs/design.md docs/decisions.md docs/tasks.md
```

各ファイルには「テンプレート」を入れておくと楽。

### CLAUDE.md は標準ガイドラインから生成する

**重要なメタルール**：

- この **標準ガイドラインは「人間が参照する大きな手引書」**
- **CLAUDE.md は「AI が読み込むプロジェクト固有の指示書」**
- 標準ガイドラインを全部 AI に読ませると、コンテキストの無駄になる
- だから、**新規プロジェクト開始時には、標準ガイドラインを参考に CLAUDE.md を生成する**

#### CLAUDE.md 生成プロンプト例

新規プロジェクトの開始時、Claude や ChatGPT に以下を投げる：

```
DEPT.405 開発ガイドライン（dept405_development_guideline.md）を参考に、
このプロジェクト（プロダクト名: XXX、目的: YYY）の CLAUDE.md を作ってください。

含めるべき内容：
- プロジェクト概要
- 必読ドキュメント
- 厳守ルール（ドキュメント先行、実装前チェックリスト など）
- 技術スタック
- ファイル構造
- セッション開始時のフロー
- 実装中のチェックポイント
```

これで、標準ガイドラインの内容が、各プロジェクトの CLAUDE.md に自動で反映される。

## 10. プロジェクト管理：GitHub の使い方

### リポジトリ構成

DEPT.405 では、以下の 2 種類のリポジトリを使い分ける：

**汎用ドキュメント用リポジトリ（1 つ）**

```
dept405-meta/
├── README.md（DEPT.405 全体の概要）
├── guidelines/
│   └── development-guideline.md（このガイドライン）
├── templates/
│   ├── CLAUDE.md.template
│   ├── requirements.md.template
│   ├── design.md.template
│   ├── decisions.md.template
```

### コミット運用のコツ（Conventional Commits）

コミットメッセージは **`prefix: 内容`** の形式で統一する。  
これでドキュメント履歴表に書かない「細かい変更」も、後から `git log` で意味がわかる。

**prefix 一覧**：

| prefix | 用途 | 例 |
|---|---|---|
| `docs:` | ドキュメント更新 | `docs: requirements.md - 半自動分類を v1.0 に含める判断追加` |
| `feat:` | 新機能実装 | `feat: 撮影画面の連続撮影機能を実装` |
| `fix:` | バグ修正 | `fix: OCR キーワード判定の優先度バグ修正` |
| `refactor:` | リファクタリング（機能は変えずに構造改善） | `refactor: db アクセス層を分離` |
| `chore:` | 雑務（依存更新、設定変更、ファイル整理など） | `chore: Tesseract.js を 5.0 に更新` |
| `test:` | テスト追加・修正 | `test: classifier の単体テスト追加` |
| `style:` | コードスタイルのみの変更（ロジック変更なし） | `style: lint 警告を解消` |

**書き方のコツ**：

- **小さなコミット**を心がける（1 つのコミット = 1 つの目的）
- メッセージは「何を変更したか」が瞬時にわかるように
- ドキュメントの大きな節目では、バージョン番号も含める  
  例: `docs: requirements.md v0.3 - 子どもの権限拡張、フォルダ共有設定追加`
