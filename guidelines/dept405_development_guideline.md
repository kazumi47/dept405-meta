## 1. なぜ Spec-Driven Development か

バイブコーディングでは「実装 → 動作確認 → 『あ、これ違う』→ 修正」のループの中で、要件定義書の更新が抜け落ちやすい。気づいたときには、ドキュメントと実物の乖離が積み重なっていて、後から見返しても「何を作ろうとしていたか」が再構成できなくなる。

SDD が採用するのは逆順の発想で、「あ、これ違う」と気づいたら**先にドキュメントを直し、それから実装に入る**。ドキュメントが「コードの設計図」であり Single Source of Truth になる、という前提を崩さない。

これは手続き上の潔癖症ではない。**AI エージェントは企画時の会話を知らない**。会話で決めたことは、文書に焼かれて初めて実装者にとって存在する。モデルを切り替えても、セッションを跨いでも、文書だけが続きを保証する。具体的な文書構成・起草→決裁の作法・バージョン凍結の運用は `skills/spec-driven-dev` に委譲した。

## 2. DEPT.405 の開発観

- **1 人 + AI**（Claude Code / Antigravity / Claude.ai を用途別に使い分け）が基本体制。人手が増えないぶん、判断の記録がそのままチームの記憶になる。
- **人間が定義する・検証する・責任を持つ。AI は実装する。** 人間の仕事は「良い」の定義（要件・DoD）と、出力の検証（レビュー・採点）と、判断の記録（決裁）。細かい指示ではなく背景とゴールを渡し、推論過程ごと検収する。
- ユーザーは元 SE だが、現在は非常勤コーチで、ノンエンジニアのバイブコーダーという立ち位置。詳細なコードを読まなくても、設計と要件で全体像を把握できることを重視する。上流工程の経験と、実装をAIに委ねる働き方の組み合わせが DEPT.405 のスタイル。
- スキル向上のため、設計フェーズは意図的に丁寧にやる。急いで実装に入ることが目的ではない。

## 3. ガイドライン逸脱の扱い（DEPT.405 の価値観）

完全に厳格にすると開発が辛くなる。かといって無視すると、ドキュメントと実物がグダグダに乖離する。

そこで DEPT.405 は一貫して**「警告するけど、強制はしない」**という立場を取る。AI はルール違反の兆候（ドキュメント未更新のまま実装・外部 API の無検討利用・テスト省略など）に気づいたら必ず確認を入れるが、最終判断はユーザーに委ねる。「今回はこのまま行く」と言われたら、AI はその判断を尊重する。

ただし逸脱は記録する。理由は、逸脱そのものを咎めるためではなく、**後で「なぜあの時そうしたか」を思い出せるようにするため**。記録の型（状況・本来の指示・実際の判断・理由）は decisions.md のフォーマットとして skills 側に定義済み。

バイブコーディングを否定しない、という立場は `security-baseline` にも通底している。ガードレールを持たせることで AI は味方になる、という考え方。

## 4. 複数プロジェクト並走という前提

DEPT.405 では複数プロジェクトが並走する。各プロジェクトは独立したリポジトリを持ち、独自の CLAUDE.md・docs・tasks.md で完結する。一方で、プロジェクトを横断する知見（型・失敗パターン・評価の作法）は個別リポジトリに閉じ込めると再利用できない。

この dept405-meta リポジトリが存在する理由はここにある。**横断的なノウハウを一元管理し、`skills/` として全プロジェクトに配布する**ことで、1 つのプロジェクトで踏んだ罠を、次のプロジェクトでは踏まずに済むようにする。

セッションを切り替える時は、前のプロジェクトの tasks.md を更新してから、新しいプロジェクトの CLAUDE.md と docs を再読み込みする——これだけは思想というより最低限の作法として明記しておく。

## 5. 学びの循環という思想

DEPT.405 の開発ループは、1 プロジェクトの完了で終わらない。

```
プロジェクト完了
  → retrospective を 1 枚書く（何を学んだか）
  → 学びを skills/ に追記する（型として汎化する）
  → git push で全プロジェクト・全環境に反映される
```

振り返りは「感想文」ではなく、**次のプロジェクトの型を更新するための入力**として書く。「今回この罠を踏んだ」「この判断は正しかった」は、個別プロジェクトの decisions.md に留めず、汎化できる部分は skills 側に転記する。これにより、skills は使うたびに育つ生きたドキュメントになる。

retrospectives/ はこの循環の記録であり、skills の改訂履歴の裏付けでもある。
